# RFE-3986: Minimum Viable URI Syntax

**Status:** Evidence-derived profile of [RFC 3986](https://www.rfc-editor.org/rfc/rfc3986).
**Derived from:** 9,266,505 real-world URIs (Common Crawl `CC-MAIN-2024-10` page
URLs + WAT outlinks, Wikipedia `enwiki` externallinks), seed `0`, prune threshold
`0.001`. Reproducible — see [§9](#9-provenance-and-reproducibility).

This document is an **original profile** of RFC 3986. It does not reproduce or
modify RFC 3986's text; it defines a narrower normative syntax and records, in
[§6](#6-deltas-from-rfc-3986-normative), exactly how and why it differs. RFC 3986
remains the authoritative reference for URI semantics (normalization, comparison,
resolution) that this profile does not restate.

The key words MUST, MUST NOT, SHOULD, and MAY are to be interpreted as described
in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119). ABNF is per
[RFC 5234](https://www.rfc-editor.org/rfc/rfc5234).

## 1. Abstract

RFC 3986 defines a generic URI syntax broad enough to cover URI forms that, in
practice, do not occur. A profile that a parser must accept can be substantially
smaller than what the grammar permits. RFE-3986 is that profile: a URI syntax in
which **every production is present because real-world usage exercised it**, or
because it is load-bearing for security or interoperability and is explicitly
protected. Syntax that RFC 3986 permits but the measured corpus did not use is
**not part of this profile**, and a conforming implementation MUST reject it.

## 2. Introduction

The generic syntax of RFC 3986 is deliberately permissive: it admits relative
references, scheme characters, host forms, and delimiter characters that a
general URI must be *able* to express. Empirically, the URIs that cross the wire
occupy a much smaller region of that grammar. Accepting the full grammar when
only the smaller region is used is attack surface without benefit — every unused
production is code a parser must carry and an input an attacker may probe.

RFE-3986 narrows the grammar to observed practice, with one deliberate exception:
a small set of security- and interoperability-critical productions are retained
even at near-zero measured frequency (see [§7](#7-criticality-overrides)), because
their absence would be a vulnerability rather than a simplification.

The syntax below is the normative grammar of this profile. It is generated

## 4. Normative syntax

A conforming URI MUST match the `URI` production, and MUST NOT exercise any
production outside this grammar. This grammar is self-closed (every referenced
rule is defined) and contains no relative-reference forms.

```abnf
; Minified MVS grammar for rfc3986-uri (generated).
; 88 nodes pruned below usage threshold 0.001.

URI = scheme ":" hier-part [ "?" query ] [ "#" fragment ]
hier-part = "//" authority path-abempty / path-rootless
scheme = ALPHA *( ALPHA / DIGIT / "-" )
authority = [ userinfo "@" ] host [ ":" port ]
userinfo = *( unreserved / pct-encoded / sub-delims / ":" )
host = IP-literal / IPv4address / reg-name
port = *DIGIT
IP-literal = "[" ( IPv6address / IPvFuture ) "]"
IPvFuture = "v" 1*HEXDIG "." 1*( unreserved / sub-delims / ":" )
IPv6address = 6( h16 ":" ) ls32 / "::" 5( h16 ":" ) ls32 / [ h16 ] "::" 4( h16 ":" ) ls32 / [ *1( h16 ":" ) h16 ] "::" 3( h16 ":" ) ls32 / [ *2( h16 ":" ) h16 ] "::" 2( h16 ":" ) ls32 / [ *3( h16 ":" ) h16 ] "::" h16 ":" ls32 / [ *4( h16 ":" ) h16 ] "::" ls32 / [ *5( h16 ":" ) h16 ] "::" h16 / [ *6( h16 ":" ) h16 ] "::"
h16 = 1*4HEXDIG
ls32 = ( h16 ":" h16 ) / IPv4address
IPv4address = dec-octet "." dec-octet "." dec-octet "." dec-octet
dec-octet = DIGIT / %x31-39 DIGIT / "1" 2DIGIT / "2" %x30-34 DIGIT / "25" %x30-35
reg-name = *( unreserved / pct-encoded )
path-abempty = *( "/" segment )
path-rootless = segment-nz
segment = *pchar
segment-nz = 1*pchar
pchar = unreserved / pct-encoded / sub-delims / ":" / "@"
query = *( pchar / "/" )
fragment = *( pchar / "/" / "?" )
pct-encoded = "%" HEXDIG HEXDIG
unreserved = ALPHA / DIGIT / "-" / "." / "_" / "~"
sub-delims = "&" / "(" / ")" / "+" / "," / ";" / "="
ALPHA = %x41-5A / %x61-7A
DIGIT = %x30-39
HEXDIG = DIGIT / "A" / "B" / "C" / "D" / "E" / "F"
```

## 5. Components

- **`scheme`** — a letter followed by letters, digits, or `-`. See
  [§6.2](#62-scheme-character-set) for why `+` and `.` are excluded.
- **`authority`** — optional `userinfo`, a `host`, and an optional `port`. All
  three are retained under the override floor ([§7](#7-criticality-overrides))
  despite low measured frequency.
  - **`userinfo`** — retained for security analysis; see
    [§8](#8-security-considerations).
  - **`host`** — a registered name, an IPv4 literal, or a bracketed IP-literal
    (IPv6 / IPvFuture). All literal host forms are retained.
  - **`port`** — decimal digits.
- **`path`** — either `path-abempty` (a `//`-authority URI, `/`-separated
  segments) or `path-rootless` (a single leading non-empty segment). The
  relative and rootless-absolute path variants of RFC 3986 are excluded
  ([§6.1](#61-relative-references)).
- **`query`**, **`fragment`** — `pchar` runs, with `/` and (for `fragment`) `?`
  permitted.
- **`pct-encoded`** — `%` followed by two hex digits; retained unconditionally
  ([§8](#8-security-considerations)).
- **Character classes** — `unreserved` includes `~`; `sub-delims` is the observed
  subset (see [§6.3](#63-sub-delimiter-character-set)).

## 6. Deltas from RFC 3986 (normative)

This section is the substance of the profile: what RFE-3986 excludes relative to
RFC 3986, and the evidence for each exclusion. A conforming implementation MUST
reject inputs that rely on an excluded production.

### 6.1 Relative references

RFC 3986 defines `URI-reference = URI / relative-ref` and the associated
`relative-part`, `path-noscheme`, and `segment-nz-nc` productions. Every URI in
the corpus was absolute (carried a scheme); the relative-reference machinery was
never exercised and is **excluded**. RFE-3986 defines only absolute URIs. An
application that must resolve relative references MUST do so against RFC 3986
before applying this profile.

### 6.2 Scheme character set

RFC 3986: `scheme = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." )`.
RFE-3986: `scheme = ALPHA *( ALPHA / DIGIT / "-" )`.

In 9,266,505 URIs the trailing scheme characters occurred as: `-` 91 times,
`DIGIT` 3 times, `.` twice, `+` never. `-` and digits are structurally standard
in real scheme names (`s3`, `u2f`, `ms-word`, `view-source`) and are **retained**;
`+` (unobserved) and `.` (2 occurrences, and not structurally standard for a
scheme name) are **excluded**. This is the profile drawing a tighter boundary
than RFC 3986 where the evidence supports it.

### 6.3 Sub-delimiter character set

RFC 3986: `sub-delims = "!" / "$" / "&" / "'" / "(" / ")" / "*" / "+" / "," / ";" / "="`.
RFE-3986: `sub-delims = "&" / "(" / ")" / "+" / "," / ";" / "="`.

`!`, `$`, `'`, and `*` each occurred in fewer than 0.05% of samples and are
**excluded**. A URI relying on one of these sub-delimiters is rejected.

### 6.4 Non-normative grouping rules

RFC 3986's `gen-delims`, `reserved`, and the umbrella `path` rule are grouping
constructs; their constituent characters survive through `sub-delims` and
`pchar`, but the grouping rules themselves were never the recognized production
and are not part of the profile. This is a representational simplification with
no effect on the accepted language.

## 7. Criticality overrides

The following productions are **retained regardless of measured frequency**
because their omission would be a security or interoperability defect, not a
simplification. A page-URL corpus systematically under-counts them. Each is
recorded, with justification and an accountable owner, in the Criticality
Override Registry ([`overrides.yaml`](../../pipeline/mvs_pipeline/overrides.yaml);
governance in [`docs/overrides.md`](../../docs/overrides.md)).

| Production | Rationale |
| --- | --- |
| `pct-encoded` | Percent-encoding MUST be parsed to normalize and reject encoded traversal (`%2e%2e`) and injection payloads. |
| `userinfo` | Recognizing `user@host` is required to defend against authority-confusion / phishing (`https://trusted.example@attacker.example`). |
| `IP-literal`, `IPv6address`, `IPv4address` | Literal-host forms MUST be parsed so SSRF and private-range controls can inspect them (`[::1]`, `169.254.169.254`). |
| `port` | Non-default ports are load-bearing for services and for same-origin identity (scheme + host + port). |
| `fragment` | Fragments carry client-side routing, media fragments, and accessibility anchors; they are never sent to servers, so crawl corpora under-count them. |
| scheme `DIGIT`, `-` | Standard scheme characters just under the pruning threshold. |

`IPvFuture` is not listed separately: retaining `IP-literal` reachably retains it
(`IP-literal = "[" ( IPv6address / IPvFuture ) "]"`).

## 8. Security considerations

- **Strict rejection, no degradation.** A conforming implementation MUST reject a
  non-conforming input outright and MUST NOT attempt to interpret it under the
  broader RFC 3986 grammar. Silent fallback to the superset would reintroduce the
  attack surface this profile removes. The reference implementation fails with a
  specific `ERR_MVS_UNSUPPORTED_NODE` naming the excluded production, never a
  best-effort parse.
- **Percent-encoding** is retained precisely so that a parser can decode and
  normalize before making a security decision; an implementation MUST NOT treat
  an un-decoded octet sequence as equivalent to its decoded form.
- **Authority confusion.** `userinfo` is retained so that the `user@host`
  boundary is recognized rather than silently folded into the host; failing to
  parse it is what enables `https://good.example@evil.example` attacks.
- **Literal hosts.** IPv4 and IP-literal (IPv6) host forms are retained so that
  SSRF and private-/link-local-range policy can be enforced on literal targets.
- **Bounds.** An implementation SHOULD enforce a maximum input length and parse
  depth and reject inputs that exceed them before full recognition; the reference
  implementation defaults to 8 KiB / depth 16384 for URIs.

## 9. Conformance

An implementation conforms to RFE-3986 if, for every input, it accepts the input
**iff** the input matches the `URI` production in [§4](#4-normative-syntax)
without exercising any production excluded by [§6](#6-deltas-from-rfc-3986-normative).
The native reference parser (`mvs-refparse`) and the `mvs-validate` tool are the
executable definition of this behavior:

```bash
mvs-validate --ast artifacts/rfc3986-uri.ast.json \
             --pruned spec/rfc3986-uri/pruned.json \
             --uri 'https://user@host.example:8443/a/b?q=1#f'   # accept
mvs-validate --ast artifacts/rfc3986-uri.ast.json \
             --pruned spec/rfc3986-uri/pruned.json \
             --uri 'foo+bar://example.com/'                     # ERR_MVS_UNSUPPORTED_NODE
```

## 10. Provenance and reproducibility

The grammar in [§4](#4-normative-syntax) is not authored by hand; it is the
mechanical result of measuring usage and pruning. Fixing the seed and the crawl
identifiers reproduces the corpus, and therefore this grammar, exactly. The
provenance block on `hits.json` and `pruned.json` ties every inclusion and
exclusion in [§6](#6-deltas-from-rfc-3986-normative) back to its evidence. The
collection and regeneration commands are in
[`spec/rfc3986-uri/README.md`](README.md) and [`docs/collector.md`](../../docs/collector.md).
